# Implementation

Created task after observing that the project REPL had optional Kaocha on the classpath, causing `scry.cli-test/run-cli-no-tests-and-runner-errors-test` to fail its unavailable-adapter expectations even though `scry/failures` itself was working.

2026-06-03 architecture review: Reviewed design.md against AGENTS.md architecture notes; META.md and doc/architecture.md are absent. No new actionable architectural-fit feedback. The design preserves the core/Kaocha optional dependency boundary, keeps `scry.kaocha/run` dynamically resolved only for requested Kaocha mode, limits the new seam to internal CLI infrastructure, and aligns with deterministic REPL-focused testing.

2026-06-03 ambiguity review: Found one actionable ambiguity. The design allows either `:resolve-kaocha-runner` or `:run-kaocha`, but does not pin the boundary contract/arity/error wrapping; this leaves unclear whether tests should preserve normal `run-kaocha` orchestration or may replace it wholesale.

2026-06-03 ambiguity follow-up: Pinned the optional Kaocha CLI boundary contract in `design.md`. The design now chooses resolver-level injection via zero-arity `:resolve-kaocha-runner`, keeps `run-kaocha` responsible for normal progress/orchestration/result handling, requires the default resolver to dynamically `requiring-resolve` `scry.kaocha/run` at execution time, and specifies FileNotFoundException wrapping into the existing `:scry.cli/runner-error` unavailable-adapter contract. Marked the review-added `design-steps.md` item complete.

2026-06-03 inconsistency review: Found one actionable design/code mismatch. `design.md` says the injected Kaocha runner accepts the normalized Kaocha option map that `run-kaocha` already builds, but `src/scry/cli.clj` currently builds normalized options before `run-kaocha`; `run-kaocha` only resolves/invokes the runner and adds `:progress-callback`. The design should align that wording with the current CLI flow so implementation does not move normalization responsibility accidentally.

2026-06-03 inconsistency follow-up: Aligned `design.md` with the current CLI flow. The resolver-level boundary now says the returned Kaocha runner accepts the already-normalized Kaocha options passed into `run-kaocha` after `run-kaocha` adds the CLI `:progress-callback`; option normalization remains owned by `normalize-exec-opts`, not `run-kaocha`. Marked the review-added `design-steps.md` item complete.
2026-06-03 plan ambiguity review: Found one actionable ambiguity. The plan requires nil or non-invokable `:resolve-kaocha-runner` returns to be runner errors and asks for focused assertions, but does not pin the stderr/message contract those assertions should expect, leaving implementers to choose between reusing `Could not load Kaocha CLI runner` or adding a distinct invalid-resolver message.

2026-06-03 plan ambiguity follow-up: Clarified `plan.md` for invalid `:resolve-kaocha-runner` returns. Nil and non-invokable resolver values should be Kaocha runner errors with message `Resolved Kaocha CLI runner is not invokable`, stderr prefixed by `scry CLI error:`, and ex-data including `{:type :scry.cli/runner-error :runner :kaocha}`; they are not argument errors. Marked the review-added `steps.md` item complete.

2026-06-03 plan inconsistency review: No new actionable inconsistency feedback. Reviewed design.md/design-steps.md, plan.md, steps.md, implementation.md, and the current CLI/Kaocha test/code boundary. The plan and steps consistently preserve resolver-level injection, normalized-option ownership, Kaocha runner-error contracts including invalid resolver returns, deterministic core unavailable-adapter coverage, optional available-adapter coverage, and final verification expectations.

2026-06-03 implementation pass: Added `:resolve-kaocha-runner` to the `scry.cli` IO boundary. The default resolver is zero-arity and dynamically calls `(requiring-resolve 'scry.kaocha/run)` only when Kaocha CLI mode executes; `scry.cli` still has no namespace-load require of `scry.kaocha`. `run-kaocha` now resolves through the boundary, validates an invokable return, attaches the CLI progress callback to the already-normalized Kaocha opts, and invokes the runner. Resolver `FileNotFoundException` preserves the existing unavailable-adapter runner-error message/data; other resolver failures become `Could not load Kaocha CLI runner`; nil/string returns become `Resolved Kaocha CLI runner is not invokable` runner errors.

Updated `scry.cli-test/run-cli-no-tests-and-runner-errors-test` so the unavailable-Kaocha scenario injects only a throwing `:resolve-kaocha-runner` and still exercises `cli/run-cli` end-to-end, independent of whether optional Kaocha is present on the process classpath. Added state-based coverage for nil and non-invokable string resolver returns. Confirmed `scry.cli-kaocha-test` uses the default resolver path for available-Kaocha behavior.
