Follow-up items were added to `design-steps.md`.

2026-06-04 inconsistency-review follow-up: Completed the two newly added design follow-up items. `design.md` now states that main-style `-m` parser errors remain process-oriented (non-zero plus human stderr only) and are intentionally outside the structured outcome-kind contract; machine callers needing argument-error classification should use `clojure -X scry.cli/run`, direct `scry.cli/run`, or option-normalization APIs. The pass/zero-tests wording now consistently says only concrete var-backed canonical entries count as executable test vars; synthetic/non-var-backed entries and aggregate pass counts cannot make an otherwise zero-executable run classify as `:scry.cli/pass`. Both review-added items in `design-steps.md` are checked.

2026-06-04 plan ambiguity review: Found two new actionable plan ambiguities. Second, the plan says only concrete var-backed entries count as executable, but it does not decide whether CLI summary `:tests` counts and `:var-count` should continue counting all canonical entries or only concrete executable vars, leaving synthetic load/unknown/zero-test stdout summaries ambiguous. Follow-up items were added to `steps.md`.

2026-06-04 plan-ambiguity follow-up: Completed the two newly added actionable follow-up items without implementing the older unchecked slices. `plan.md` now makes `:scry.cli/outcome-kind` authoritative for CLI `:exit-code` (`:scry.cli/pass` => 0; every other kind => non-zero), explicitly requiring implementation/tests to avoid the old total-entry `var-count` pass path for synthetic-only passing entries. It also chooses to preserve existing human summary compatibility: `:summary :tests` and `:var-count` continue counting all canonical result entries, including synthetic/non-var-backed entries, while outcome classification separately counts only concrete var-backed executable entries.
