2026-06-04 ambiguity review: Found five actionable ambiguities in the design. The design needs to pin the exact CLI outcome classification API/key/vocabulary, mixed-signal precedence, how to identify synthetic load/suite errors from canonical results, nil-var progress/result-file naming details and collision behavior, and which public docs must document the machine-readable classification contract. Follow-up items were added to design-steps.md.

2026-06-04 ambiguity-review follow-up: Completed all newly added design follow-up items. `design.md` now pins the additive CLI classification contract as top-level `:scry.cli/outcome-kind` on `run-cli` outcomes and `clojure -X` non-zero ex-data, including success outcomes; defines initial stable pre-1.0 vocabulary (`:scry.cli/pass`, `:scry.cli/argument-error`, `:scry.cli/runner-error`, `:scry.cli/load-error`, `:scry.cli/test-failure`, `:scry.cli/unknown-result`, `:scry.cli/zero-tests`); specifies mixed-signal precedence; defines synthetic load/suite errors structurally as failing/erroring canonical entries without a concrete namespaced var symbol after a valid `:canonical-results` vector exists; pins per-status synthetic progress/result-file tokens and collision handling; and names README.md, SKILL.md, AGENTS.md, and CHANGELOG.md as the docs to update when implemented. All review-added items in `design-steps.md` are checked.

2026-06-04 inconsistency review: Found two actionable design inconsistencies. The design's precedence includes `-m` parser errors in outcome classification while the public structured surfaces only name `run-cli` outcomes and `clojure -X` ex-data, and the `:pass`/`:zero-tests` contract says executable concrete vars while the pass fallback is phrased in terms of computed exit code. Follow-up items were added to `design-steps.md`.

2026-06-04 inconsistency-review follow-up: Completed the two newly added design follow-up items. `design.md` now states that main-style `-m` parser errors remain process-oriented (non-zero plus human stderr only) and are intentionally outside the structured outcome-kind contract; machine callers needing argument-error classification should use `clojure -X scry.cli/run`, direct `scry.cli/run`, or option-normalization APIs. The pass/zero-tests wording now consistently says only concrete var-backed canonical entries count as executable test vars; synthetic/non-var-backed entries and aggregate pass counts cannot make an otherwise zero-executable run classify as `:scry.cli/pass`. Both review-added items in `design-steps.md` are checked.

2026-06-04 plan ambiguity review: Found two new actionable plan ambiguities. First, the plan adds `:scry.cli/outcome-kind` classification but does not explicitly make that classification authoritative for `:exit-code`; the current `exit-code` logic still uses total canonical entry count as `var-count`, so a synthetic-only passing entry could classify as zero-tests by design while still exiting 0 unless the implementation slices are pinned to use concrete executable var-backed entries/outcome kind for exit decisions. Second, the plan says only concrete var-backed entries count as executable, but it does not decide whether CLI summary `:tests` counts and `:var-count` should continue counting all canonical entries or only concrete executable vars, leaving synthetic load/unknown/zero-test stdout summaries ambiguous. Follow-up items were added to `steps.md`.

2026-06-04 plan-ambiguity follow-up: Completed the two newly added actionable follow-up items without implementing the older unchecked slices. `plan.md` now makes `:scry.cli/outcome-kind` authoritative for CLI `:exit-code` (`:scry.cli/pass` => 0; every other kind => non-zero), explicitly requiring implementation/tests to avoid the old total-entry `var-count` pass path for synthetic-only passing entries. It also chooses to preserve existing human summary compatibility: `:summary :tests` and `:var-count` continue counting all canonical result entries, including synthetic/non-var-backed entries, while outcome classification separately counts only concrete var-backed executable entries. Synthetic load/unknown entries therefore appear in stdout `Tests:` counts as result entries, and an empty canonical vector prints zero test counts while classifying/exiting as zero-tests. The review-added items in `steps.md` are checked.
